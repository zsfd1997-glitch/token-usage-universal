# GPT 画图提示词：对话如何塑造 Token Usage Universal

## 主提示词

```text
请为我生成一张高完成度、强叙事感、信息密度很高的“战略信息图海报 / editorial infographic poster”，主题是：

《一场对话，如何把一个本地 token 统计工具，硬生生逼成一个真正的产品》

这不是普通的聊天记录拼贴，也不是软件宣传海报，而是一张“对话驱动产品演化”的视觉分析图。核心意思是：
这个项目不是从抽象架构自然长出来的，而是被一轮轮真实对话不断推进、纠偏、验收、冻结约束，最终被逼成型。

整张图要同时传达 4 件事：
1. 这批对话不是闲聊，而是高压推进。
2. 需求不是慢慢堆出来的，而是被不断纠偏后收敛出来的。
3. 项目从一个较小的本地工具，被逼成了 Top20 生态、跨 Desktop / CLI / IDE、local-first / exact-first / source-first 的产品。
4. 对话本身，就是产品治理过程。

请采用“冷静判断 + 热辣推进 + 略带叛逆”的视觉气质：
- 不是可爱卡通
- 不是企业 PPT 模板
- 不是普通 dashboard 截图拼贴
- 要像顶级科技杂志的专题视觉、未来感编辑设计、战略情报图
- 有压迫感、速度感、决策感
- 视觉上让人一眼就觉得：这不是在讨论，这是在逼产品进化

画面形式：
- 纵向 4:5 海报构图，适合封面图、报告头图、社交媒体长图
- 整体是“一张图读懂”的复合信息图
- 允许少量大字标题和少量标签，但不要塞满小字
- 中文文字尽量少而准，只保留大标题、少量核心标签、少量数字
- 所有文字必须大、清晰、可读，绝对不要密密麻麻小字

推荐整体构图：

第一层：中心主视觉
- 画面中心是一个被高强度对话驱动的“产品核心熔炉 / pressure core / decision engine”
- 中心可视化为一个发光的、被多条对话箭头和信号流击中的核心体
- 核心体不是球体科幻 cliché，而更像一个高密度的结构中枢、产品引擎、发热中的决策核心
- 中心只放一句最核心的大字：
  “产品收敛 + 口径纠偏 + 验收加压”
- 这句必须像整张图的发动机

第二层：围绕中心的 5 个主题扇区
请围绕中心做成半环形或放射状结构，用 5 个主要主题模块表示对话真正推动项目的方向：
- 推进续跑 51
- 产品形态与范围 34
- 覆盖适配与兼容 32
- 验证发布与交付 24
- 输出协议与文档风格 18

每个主题扇区要有：
- 一个鲜明图标或视觉隐喻
- 一个醒目的数字
- 一个非常短的中文标签

视觉隐喻建议：
- 推进续跑：连续前冲箭头、推进波、重复命令脉冲
- 产品形态与范围：文件夹分裂为“个人版 / 团队版”、skill + CLI 双形态、产品边界框
- 覆盖适配与兼容：从单点工具扩张为多个客户端节点，形成生态网络
- 验证发布与交付：Windows 压缩包、GitHub/Gitee、交付箱、测试门禁
- 输出协议与文档风格：ASCII-HIFI 终端感、README / SKILL.md、文字风格锤炼

第三层：一条清晰的时间线
在画面下半部做一条强叙事时间线，表现这个项目被对话一步步掰弯的 5 个阶段：
- 独立成包与分叉
- Top20 扩张与架构冻结
- 验证补强与防漏
- 产品化与文档契约
- 复盘与知识化

时间线不能只是五个平铺方块，要做成“不断升级、压力不断上升”的演化结构：
- 前面更像小工具
- 中段快速扩张、节点爆炸
- 后段出现验证门禁、文档契约、知识沉淀
- 时间线应该让人感觉项目越来越重、越来越硬、越来越像产品

第四层：对话动作分析
请把“你是怎么推动项目的”也可视化，不只是主题分类。
在画面一侧加入一个“对话动作面板 / dialogue force panel”，显示以下动作类别和气质：
- 推进型 54
- 纠偏型 15
- 验收型 10
- 冻结约束型 8
- 外部求证型 30

这部分不要做成普通柱状图，可以更有设计感：
- 像战术控制面板
- 像作战指令仪表
- 像高压项目管理信号灯

其中“继续”这个词出现 23 次，要被特别强调。
要让人感受到：
- 这不是用户在闲聊
- 这是一个不断给系统施压、让系统别停、别糊弄、别偷懒的推动模式

第五层：项目结果可视化
请在画面右侧或底部，让“最终被逼出来的产品原则”形成结果区，表现为最终沉淀出的 4 条原则：
- local-first
- exact-first
- source-first
- distribution-first

这些结果不要只写成四行文字。
要做成“由对话压力锻造出的最终结构件 / design laws / operating principles”的感觉。
像是一套最后成型的产品法则，从混乱对话中结晶出来。

数字信息必须明确出现：
- 24 个会话
- 151 条用户原话
- “继续” 23 次
- 推进续跑 51
- 产品形态与范围 34
- 覆盖适配与兼容 32
- 验证发布与交付 24
- 输出协议与文档风格 18

建议标题层级：
- 主标题：一场对话，如何把一个本地 token 统计工具，逼成真正的产品
- 副标题：Token Usage Universal Dialogue Classification Analysis
- 核心标语：这不是边聊边做，这是边验收边重写项目定义

视觉风格要求：
- 高级、锋利、克制，不要俗艳
- 强烈的信息设计感
- 层次复杂但秩序清楚
- 颜色要有张力：炽热橙红 / 工业红 / 机械银灰 / 深石墨黑 / 冷白高光
- 避免紫色主导
- 避免幼稚插画风
- 避免扁平化 SaaS 官网模板风
- 可以带少量网格、脉冲线、结构图、界面残影、命令行感、节点网络
- 可以把用户原话做成少量大型引语碎片漂浮在图中，像对产品施压的信号

可选少量引语碎片，要求大而少，不要多：
- “我要的是全模型、全环境通用！”
- “不要只想着 codex 和 claudecode”
- “我们没有 host 环境，你想办法解决”
- “为什么没有按照 ascii-hifi 严格输出”
- “检索一下我和你的所有对话，把这个复盘充实”

这些引语不要整齐排版成正文，而要像“打在系统上的冲击波”。

画面氛围：
- 不是温和复盘
- 是高压、高密度、强执行、强纠偏
- 但依然要理性、清晰、有审美
- 让观众一眼读到：这个产品是被现实需求和真实验收一锤一锤打出来的

画质要求：
- 超高细节
- 干净锐利
- 专业排版感
- 适合做报告封面或长图头图
- 不要 UI 截图拼盘，不要杂乱无章，不要文字糊掉

请输出一张完成度极高、可直接用作封面级配图的作品。
```
