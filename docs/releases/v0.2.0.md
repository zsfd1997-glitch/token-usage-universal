# Token Usage Universal v0.2.0

`v0.2.0` 这一版，核心不只是再加几个零散命令，而是把 `token-usage-universal` 从“会统计 token 的 CLI”继续推成一个真正能直接看的终端控制面，而且默认口径也比早期版本更接近“真正活跃消耗”。

现在，您跑一次 `report --today`，就不只是拿到一个数字，而是能在一屏里看到：

- 今天总量
- 去缓存后 token
- 当前会话
- 按模型 / 按项目拆分
- 最近趋势
- 月历热力图
- 哪些来源没统计到，以及为什么没统计到

整个方向没变，还是这三条：

- `local-first`
- `exact-first`
- `source-first`

## Highlights

### 1. 默认报告升级成有效 token 导向的终端控制面

这版最大的变化，是默认 `report` 输出不再只是命令行明细，而是一个完整 dashboard，而且默认主口径切到了 `去缓存后 token`。

对短触发场景，比如“今天用了多少”“最近怎么样”，现在默认会直接给出：

- 总览
- `总 token / 去缓存后 token` 区分
- 当前会话
- 按模型
- 按项目
- 最近 7 天 / 30 天趋势
- 月历热力图
- 诊断 / 缺失来源提醒

这样做的目的很简单：
以前您得自己再追问“按模型看看”“按项目看看”“为什么没统计到”，现在默认先把最常用的信息摆出来。与此同时，默认费用也改成按去缓存后 token 估算，不再把缓存命中直接混进主口径。

### 2. 新增一批可直接展开看的视图

`v0.2.0` 新增或补强了下面这些视图能力：

- `--by day`
- `--trend 7d`
- `--trend 30d`
- `--calendar month`
- `--current-session`
- `--session <id>`

对应的体验是：

- 想看最近是不是暴涨了，可以直接看趋势
- 想看今天火力主要花在哪个项目，可以直接看项目拆分
- 想看眼前这一轮对话烧了多少，可以直接看当前会话卡片
- 想查某个特定 session，可以直接点名看会话详情

### 3. `explore` 和“最近 token”交互更收敛

如果在 TTY 里使用，现在可以直接走 `explore` 入口弹出视图选择，而不一定非要先记住全部参数。

这版把 `explore` 入口收敛成更直白的短选项：

- 今天总览
- 最近一周 token 消耗情况
- 最近一个月 token 消耗情况
- 按模型看今天
- 按项目看今天
- 当前会话
- 关闭

这版的重点不是做花哨交互，而是把“我就想快速看看”这件事做顺手。

## 4. 常见视图被真正压快了

这版不只是多了图，还把常见路径真的压快了：

- 新增文件级增量缓存，未变更日志不再重复 JSON 解析
- 新增按天预聚合索引，`today / recent / 7d / 30d / 月历` 常见视图优先走轻量日级路径
- `Codex` 不再每次整棵树 `rglob`，而是优先按日期目录定点扫描当前窗口相关天数
- `PricingDatabase` 改成进程内单例复用，不再每次重复读价格库

在本机热缓存验证里：

- `report --today` 约为 `0.26s`
- `report --last 30d --dashboard recent` 约为 `0.26s`

冷启动仍然要读本地大日志、顺手建索引，所以首轮会更重；但重复查看常见面板时，已经进入近乎秒开状态。

## 5. 新增本地价格库和 `estimated cost`

这版加入了本地价格库，可以在报告里显示 `estimated cost`，而且最近趋势和本月分布右侧也补上了 `合计 + 估算费用 / 均值 / 最高` 这组摘要。

但这件事有一个边界，我们在产品口径上写得很死：

- 它只是本地价格表估算
- 它不是账单真源
- 它不能覆盖 `exact` token 结论

也就是说：
它适合回答“今天大概花了多少钱”，不适合冒充云账单或供应商结算结果。

## 6. 诊断能力前移，不再静默漏掉来源

`v0.2.0` 另一个很实际的改进，是把“为什么没统计到”前移到了默认体验里。

现在如果某个来源没被算进去，默认 dashboard 会直接给出：

- 缺了什么真源
- 为什么当前还拿不到 exact 数据
- 下一步要配哪个路径或环境变量

例如：

- `claude-code` 只有 transcript，或者只有新版 session-config 文件、没有带 `total_tokens + executor_end/grader_end` 的 exact JSON
- `generic-openai-compatible` 还没配置 `TOKEN_USAGE_GENERIC_LOG_GLOBS`

这样您不用再看到一个“空结果”，然后反过来猜到底是没用，还是没统计到。

## 7. Claude 和 Windows 兼容进一步补齐

这一轮还把跨平台和多布局兼容补到了更能落地的程度：

- `claude-code` 不再死认 `timing.json`，而是接受任意带 `total_tokens + executor_end/grader_end` 的 Claude exact JSON
- 如果目录里只有 `.claude.json / cowork_settings.json / manifest.json` 这类 session-config 文件，会明确诊断为“有布局、没 token 真源”
- Windows 下的 `%USERPROFILE% / %APPDATA%` 环境变量占位现在可以展开
- Windows 风格项目路径现在也能正确提取项目名
- Claude 的 macOS / Windows 默认真源路径和说明矩阵已经同步到 README / INSTALL / SKILL / source-contract

## 8. 验证覆盖继续加厚

这版测试已经扩到 `66` 个，覆盖范围比早期 `v0.2.0` 草稿更完整，重点补在这些地方：

- 聚合逻辑
- 本地价格库
- 图形 renderer
- CLI 集成
- 文件级缓存与并发写盘
- 按天预聚合和 recent 30d 快路径
- Windows 路径与环境变量兼容
- Claude 多版本 exact JSON 和新版布局 diagnose
- dashboard 默认输出
- 当前会话 / 会话详情
- 趋势图 / 月历热力图
- 缺失来源诊断

## Compatibility

对现有使用者来说，这版不是重做，而是增强。

保持不变的部分：

- 还是 `Codex skill`
- 还是内部 `Python CLI core`
- 还是 `report / sources / health / diagnose` 这套主入口
- `Task Master` 依然只是可选规划 handoff，不是运行时依赖

所以如果您已经在用 `v0.1.0`：
这版主要是“更好看、更好查、更少追问”，不是强迫您迁移使用方式。

## Upgrade Notes

- 这版没有强制迁移动作。
- 原有命令仍然可用。
- 如果您要接 `generic-openai-compatible`，仍然需要显式设置 `TOKEN_USAGE_GENERIC_LOG_GLOBS`。
- 如果您看到 `estimated cost`，请把它理解为“本地估算值”，不要把它当供应商账单。
- 如果您之前看过早期 `v0.2.0` 草稿，请以这版为准；测试数、Claude 诊断范围和快路径性能数据都已经刷新。
